# ADR-0005: RPO / RTO для Cutover 1

**Status:** accepted (2026-05-30)

## Context

Cutover 1 (Этап 1, итерация 10) переводит production-трафик BillHub со старой VPS + Supabase Cloud + Cloudflare R2 на новую VPS + Yandex Managed PostgreSQL + Cloud.ru S3.

Бизнес-требования:
- Cutover планируется на нерабочее время (выходные, ночь).
- Допустимое окно обслуживания — несколько часов, но чем короче, тем лучше.
- Потеря данных (по успешно подтверждённым write-операциям) недопустима.

## Decision

### RPO (Recovery Point Objective)

**RPO = 0.** Никаких потерь подтверждённых write-операций.

Достигается тем, что:
- В момент начала cutover-окна старый прод переводится в read-only (см. [ADR-0003](0003-cutover-db-strategy.md)). Write-операции пользователей физически невозможны.
- Между моментом read-only и моментом `pg_dump` Supabase данные не меняются → dump содержит ВСЕ подтверждённые операции до окна.
- Финальная `rclone sync --update` (см. [ADR-0004](0004-cutover-files-strategy.md)) забирает все файлы, появившиеся между T1 (первичный copy за дни до cutover) и моментом read-only.

Если cutover откатывается (см. [ADR-0006](0006-rollback-procedure.md)) — RPO зависит от состояния:
- Если откат **до** DNS-switch — RPO = 0 (Supabase ещё read-only, новых данных в Yandex PG нет).
- Если откат **после** DNS-switch — пользователи уже могли сделать write в Yandex PG. Эти write-операции переносятся обратно в Supabase через явный скрипт (delta-export-and-replay). RPO остаётся 0 при условии успешного replay.

### RTO (Recovery Time Objective)

**RTO = 2–4 часа** для запланированного cutover. Это окно обслуживания, в которое портал полностью недоступен для write (read остаётся через maintenance-страницу).

Декомпозиция окна:

| Шаг | Время (ожидаемое) | Время (worst case) |
|---|---|---|
| 0. Уведомление пользователей + перевод в read-only | 5 мин | 10 мин |
| 1. Финальный `pg_dump` Supabase | 5–15 мин | 30 мин (если БД >20 GB) |
| 2. `pg_restore` в Yandex PG (parallel -j 4) | 10–20 мин | 45 мин (если БД >20 GB) |
| 3. Скрипт `import-passwords.ts` | 1–2 мин | 5 мин |
| 4. Финальная `rclone sync --update` файлов | 2–10 мин | 30 мин (если объём дельты неожиданно большой) |
| 5. `rclone check --size-only` + manifest verify | 2–5 мин | 10 мин |
| 6. Production startup checks на новой VPS | 1–2 мин | 5 мин |
| 7. Smoke на временном домене (Playwright короткий) | 10–15 мин | 30 мин (если есть выявленные ошибки) |
| 8. DNS cutover + propagation | 1–5 мин | 30 мин (если кэш DNS не очистился; TTL заранее снижен до 60 с) |
| 9. Smoke через основной домен | 5–10 мин | 20 мин |
| 10. Снятие maintenance | 1 мин | 5 мин |
| **Total** | **~45–90 мин** | **~3.5 часа** |

Buffer 30+ минут на непредвиденное → целевое окно 2 часа, заявляется пользователям как 4 часа.

### RTO для rollback

В случае rollback (см. [ADR-0006](0006-rollback-procedure.md)):
- Возврат DNS на старую VPS — 1–5 мин + propagation (TTL 60 с).
- Перевод старой VPS из read-only в read-write — 1 мин.
- Smoke на старой VPS — 5–10 мин.
- **Rollback RTO = 15–30 мин** при условии, что rollback инициирован в пределах cutover-окна (старая инфра ещё не очищена).

Если rollback инициирован через дни после cutover (например, обнаружена критическая проблема позже) — это уже отдельная процедура с возможной потерей write-данных, накопленных на Yandex PG (см. ADR-0006, delta-replay).

### Условия отмены cutover (без выхода в rollback)

Cutover откладывается **до начала маневров** (то есть до шага 1) при:
- Yandex PG недоступен (health-check fail).
- Cloud.ru S3 недоступен.
- Новая VPS не отвечает на `/health/ready`.
- В preflight-чек обнаружены незакоммитченные изменения или красный CI на main.

В этом случае окно обслуживания не открывается, пользователи не получают maintenance-страницу. Cutover переносится.

## Consequences

**Плюсы:**
- Чётко документированный таймлайн.
- RPO 0 — критично для финансового портала (счета, оплаты, согласования).
- RTO 2–4 ч укладывается в стандартное «ночное окно».
- Worst case (3.5 ч) + buffer = заявленный 4 ч.

**Минусы:**
- Пользователи не могут делать write в течение 2+ часов.
- В случае серьёзных проблем после DNS-switch — нужно либо чинить на новой инфре (выход за RTO), либо инициировать rollback с delta-replay (потенциально сложная процедура).

## Alternatives

- **RTO 1 час (более агрессивно)** — отвергнут: не оставляет buffer на непредвиденное, smoke сокращается до неприемлемо короткого.
- **RTO 8+ часов** — избыточно для нашего объёма данных; растягивает downtime без выигрыша.
- **Zero-downtime cutover через logical replication + blue/green DNS** — рассматривается как резервный сценарий, если объём БД вырастет >50 GB (см. [ADR-0003](0003-cutover-db-strategy.md)). На текущем объёме — overkill.
